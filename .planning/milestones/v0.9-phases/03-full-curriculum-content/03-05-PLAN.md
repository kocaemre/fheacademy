**Lesson 4.1 — Gas Optimization for FHE** (conceptual):
- learningObjective: "Understand FHE operation costs relative to standard EVM operations and apply optimization strategies to reduce gas consumption."
- Content: FHE ops are significantly more expensive than standard Solidity, cost hierarchy (euint8 cheapest through euint256 most expensive, multiplication > addition), optimization strategies (minimize FHE ops, use plaintext where possible like FHE.add(encrypted, 1), batch operations, smallest type, avoid re-encryption)
- CodeDiff: Unoptimized FHEVM contract (using euint256, multiple redundant ops) vs optimized version (euint32, batched ops, plaintext constants)
- CodeBlock: Gas cost reference table showing relative costs per type and operation
- CalloutBox (tip): Use plaintext constants where possible — FHE.add(encrypted, plaintext) is cheaper than FHE.add(encrypted, FHE.asEuint32(plaintext))
- CalloutBox (warning): Every FHE operation consumes significant gas — think carefully before adding operations
- Quiz: (1) Which encrypted type has the lowest gas cost? (euint8), (2) Why is FHE.add(x, 1) cheaper than FHE.add(x, FHE.asEuint32(1))? (plaintext operand avoids encryption), (3) Code-reading: identify the gas optimization opportunity in given code
- InstructorNotes: Students often over-encrypt. The key insight: encrypt ONLY what needs privacy, leave everything else as plaintext.

**Lesson 4.2 — Security Best Practices** (conceptual):
- learningObjective: "Conduct security reviews of FHEVM contracts using a comprehensive audit checklist and identify common vulnerability patterns."
- Content: FHEVM-specific security checklist (6 items: ACL on every state change, overflow protection, FHE.isInitialized checks, proper externalEuint usage, secure decryption handling, no sensitive data in events/reverts), side-channel considerations (transaction patterns can leak info), re-encryption attacks
- CodeDiff: Insecure FHEVM contract (missing ACL, no overflow protection, sensitive data in events) vs secure version (all checks in place)
- CodeBlock: Complete security audit checklist as code comments
- CalloutBox (mistake): Emitting events with encrypted values — the handle is visible, and transaction patterns reveal information
- CalloutBox (warning): Side-channel: even if data is encrypted, transaction frequency and gas usage can reveal information
- Quiz: (1) Why is emitting events with encrypted data dangerous? (2) Code-reading: given a contract, identify all security issues, (3) What does FHE.isInitialized prevent?
- InstructorNotes: Frame this as a real audit exercise. Have students review each other's Week 2 homework for security issues.

**Lesson 4.3 — Confidential DeFi Concepts** (conceptual):
- learningObjective: "Design confidential DeFi protocols using FHEVM primitives, including encrypted AMMs, lending protocols, and order books."
- Content: Confidential AMM (encrypted reserves, front-running immune), confidential lending (hidden collateral), encrypted order books (orders encrypted, matching on encrypted data), private stablecoins (hidden balances with compliance), why these are IMPOSSIBLE without FHE
- CodeDiff: Standard AMM swap function (public reserves, vulnerable to MEV) vs confidential AMM concept (encrypted reserves, front-running immune)
- CodeBlock: Pseudocode for encrypted order book matching
- CalloutBox (info): Confidential DeFi is an entirely new design space — not just "adding privacy" to existing DeFi
- CalloutBox (tip): The key insight: FHE enables computation on encrypted data. ZKP can prove things about data but cannot compute on it.
- Quiz: (1) How does encrypted AMM prevent front-running? (2) Why can't ZKP alone enable confidential AMM? (3) What advantage do encrypted order books have over traditional ones?
- InstructorNotes: These are cutting-edge concepts. Students should understand the design space, not memorize implementations. This is about inspiring future projects.

**Lesson 4.4 — Testing Strategies** (hands-on):
- learningObjective: "Write comprehensive test suites for FHEVM contracts covering functional, permission, edge case, and integration testing categories."
- Content: Test categories (functional, permission, edge cases, integration), mock mode vs testnet testing differences, CI/CD with GitHub Actions, testing ACL (verify unauthorized decrypt fails), testing overflow (verify FHE.select guard works)
- CodeDiff: Basic unit test vs comprehensive FHEVM test suite (showing permission tests, edge cases, multi-user scenarios)
- CodeDiff: Standard assertion vs FHE-specific assertion (decrypt then compare)
- CodeBlock: Complete test file with all categories for a confidential token
- CalloutBox (tip): Always test what happens when ACL is wrong — it is the most common bug source
- CalloutBox (info): In CI/CD, use mock mode — it is deterministic and fast
- Quiz: (1) Why test with unauthorized addresses? (2) How do you assert an encrypted value equals an expected plaintext? (3) Code-reading: given a test, identify the missing edge case
- InstructorNotes: Testing FHE contracts is different from standard Solidity. The handle-based approach requires new mental models.

**Lesson 4.5 — Testnet Deployment** (hands-on):
- learningObjective: "Deploy an FHEVM contract to Ethereum Sepolia testnet, configure ZamaEthereumConfig, and interact with the deployed contract."
- Content: Sepolia testnet setup, deployment script configuration, ZamaEthereumConfig for testnet (automatic coprocessor setup), deployment and verification, interacting with deployed contract via SDK, "Congratulations — you can now deploy confidential smart contracts!"
- CodeDiff: Standard Hardhat deploy script vs FHEVM deploy script (showing ZamaEthereumConfig inheritance makes it mostly the same)
- CodeDiff: Local mock interaction vs testnet interaction (async decryption, real gas costs)
- CodeBlock: Complete deployment script
- CodeBlock: Post-deployment interaction script (TypeScript)
- CalloutBox (tip): ZamaEthereumConfig handles all network configuration — you just inherit and deploy
- CalloutBox (warning): Testnet FHE operations take real time (seconds to minutes for decryption). Plan for async patterns.
- Quiz: (1) What does ZamaEthereumConfig configure automatically? (2) What is the main difference between mock and testnet decryption? (3) Code-reading: given a deployment script, identify what would need to change for mainnet
- InstructorNotes: This is the culmination lesson. Students should feel accomplished. Celebrate their progress from zero FHE to testnet deployment.
