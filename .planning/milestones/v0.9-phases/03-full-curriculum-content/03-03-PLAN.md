- CodeBlock: Complete type catalog as a reference table
- CalloutBox (mistake): Using euint256 for everything — massive gas waste
- CalloutBox (tip): Trivially encrypted values are visible in calldata — not truly private
- Quiz: (1) Which type for a yes/no vote? (ebool), (2) What does FHE.asEuint32(42) do? (trivial encryption), (3) Code-reading: given a contract with wrong types, identify the issue
- InstructorNotes: Interactive exercise — 5 data types, pick the encrypted type

**Lesson 2.2 — Operations on Encrypted Data** (hands-on):
- learningObjective: "Use all arithmetic, comparison, and bitwise FHE operations, and understand the critical paradigm shift from if/else to FHE.select."
- Content: Arithmetic ops (add, sub, mul, min, max, neg), division/remainder (plaintext divisors only), comparison ops (eq, ne, lt, le, gt, ge — return ebool), bitwise ops, THE CRITICAL PARADIGM SHIFT: no if/else with encrypted conditions — use FHE.select
- CodeDiff: Standard Solidity conditional logic (if/else) vs FHEVM FHE.select pattern — this is the MOST IMPORTANT CodeDiff in the entire curriculum
- CodeDiff: Standard arithmetic operations vs FHE arithmetic operations
- CodeDiff: Standard comparison with require() vs FHE comparison returning ebool
- CalloutBox (warning): Both branches of FHE.select are ALWAYS computed — no short-circuit
- CalloutBox (mistake): Dividing by an encrypted value — only plaintext divisors supported
- Quiz: (1) Why can't you use if(ebool)? (2) Code-reading: given FHE.select code, what does it return? (3) What type does FHE.gt return?
- InstructorNotes: The paradigm shift from imperative to functional-style branching, shift operator modulo behavior

**Lesson 2.3 — Encrypted Inputs and ZKPoK** (hands-on):
- learningObjective: "Understand the complete encrypted input lifecycle and implement functions that securely accept encrypted user inputs with ZKPoK validation."
- Content: Full lifecycle (client encrypts -> ZKPoK generated -> calldata sent -> FHE.fromExternal verifies), why ZKPoK matters (replay prevention, contract binding), multi-input functions, error handling
- CodeDiff: Standard function with plaintext parameter vs FHEVM function with externalEuint + inputProof
- CodeDiff: Standard multi-parameter function vs FHEVM multi-input function (multiple externalEuint params sharing one inputProof)
- CodeBlock: Client-side SDK code for creating encrypted inputs (TypeScript)
- CalloutBox (info): The ZKPoK binds the ciphertext to a specific contract and sender
- Quiz: (1) What does FHE.fromExternal do? (2) Why can't you reuse someone else's ciphertext? (ZKPoK binding), (3) Code-reading: spot the missing inputProof parameter
- InstructorNotes: Students often confuse externalEuint with euint — external is user input, euint is internal state

**Lesson 2.4 — Access Control List (ACL) System** (hands-on):
- learningObjective: "Master the FHEVM ACL system — the most critical concept for building secure confidential smart contracts."
- Content: Core concept (every ciphertext has an ACL), FHE.allowThis vs FHE.allow, persistent permissions, permission scenarios (token balance, auction bid, multi-sig), common ACL bugs
- CodeDiff: Token transfer showing complete ACL flow — before (no ACL needed in Solidity) vs after (multiple allow calls in FHEVM)
- CodeDiff: Multi-party permission scenario — escrow contract where both parties need access
- CodeBlock: Complete ACL function reference
- CalloutBox (warning): Forgetting FHE.allowThis after state change = contract loses access to its own data
- CalloutBox (mistake): Not updating permissions after state changes — old handle permissions do NOT transfer to new handles
- Quiz: (1) What happens if you forget FHE.allowThis? (2) Code-reading: given a transfer function, identify the missing allow call, (3) When is FHE.allow needed vs FHE.allowThis?
- InstructorNotes: ACL is #1 source of bugs. Interactive exercise: draw ACL table through a transfer() step by step.

**Lesson 2.5 — Patterns and Best Practices** (conceptual):
- learningObjective: "Apply defensive programming patterns for FHEVM including overflow protection, initialization checks, and safe transfer patterns."
- Content: Overflow protection with FHE.select, FHE.isInitialized() checks, safe transfer pattern, defensive programming summary (5 rules)
- CodeDiff: Unsafe transfer (no overflow check) vs safe transfer with FHE.select guard
- CodeBlock: Complete safe transfer pattern with all ACL calls
- CodeBlock: Initialization check pattern
- CalloutBox (tip): The 5 defensive programming rules for FHEVM
- Quiz: (1) How does FHE.select prevent overflow? (2) Code-reading: given a safe transfer, what happens if balance is insufficient? (3) Why check FHE.isInitialized?
- InstructorNotes: These patterns should become automatic — students should be able to write safe transfers from memory
